构建第一个自定义方言（Dialect）

使用ODS（Operation Definition Specification）定义操作、类型和属性

📚 核心概念

方言（Dialect）是MLIR中模块化、可扩展的抽象单元，它将相关的操作、类型和属性组织在一起，形成特定领域或抽象层次的中间表示。

ODS（Operation Definition Specification）是一种基于TableGen的声明式规范，用于定义MLIR中的操作、类型和属性，它极大地简化了定义过程并自动生成C++代码。

为什么使用ODS？

  • 声明式编程：只需描述“是什么”，无需编写重复的样板代码。
  • 自动代码生成：自动生成操作验证、打印、解析等C++代码。
  • 一致性保证：确保操作定义符合MLIR框架规范。
  • 易于维护：集中管理定义，修改一处即可同步更新。

🛠️ 定义步骤详解

  1. 创建方言类：使用ODS定义方言的元数据，如名称、描述、命名空间等。
  2. 定义操作（Operation）：指定操作名、参数、结果、特性、接口等。
  3. 定义类型（Type）：创建方言特有的数据类型。
  4. 定义属性（Attribute）：定义操作附带的编译期已知属性。
  5. 生成与注册：通过TableGen生成C++代码，并在MLIR上下文中注册方言。

ODS定义表示例

// 定义方言
def MyDialect : Dialect {
let name = "my";
let summary = "我的自定义方言";
let description = [{}];
let cppNamespace = "::my";
}

// 定义操作
def MyOp : Op<MyDialect, "myop", [Pure]> {
let summary = "我的自定义操作";
let arguments = (ins F32Tensor:$input, I32Attr:$value);
let results = (outs F32Tensor:$output);
let assemblyFormat = "$input `,` $value attr-dict `:` type($input) `->` type($output)";
}
提示： ODS中的ins表示输入操作数，outs表示输出结果，attr-dict用于属性字典的打印与解析。

📊 操作、类型、属性对比

组件 作用 示例
操作（Operation） 表示计算或变换的基本单元，如加法、卷积等。 my.add, my.conv2d
类型（Type） 定义数据的类型，可以是标量、张量或自定义类型。 !my.tensor<f32>, !my.custom_type
属性（Attribute） 编译期已知的常量值，附加到操作上。 value = 42 : i32, padding = dense<0>

关键ODS构造块

  • OpInterface：为操作添加通用接口，如InferTypeOpInterface。
  • Traits：定义操作特性，如Commutative、Pure。
  • AssemblyFormat：自定义操作的文本格式，提升可读性。
  • Verifier：自动生成或自定义验证逻辑，确保操作合法性。

🎯 工程实践指南

在AI编译器工程中，自定义方言通常用于：

  • 领域特定抽象：如神经网络层、硬件原语等。
  • 中间优化表示：在高级方言与低级方言之间引入中间表示。
  • 硬件后端支持：为特定加速器定义专有操作和类型。

最佳实践

  1. 命名清晰：使用有意义的方言前缀，避免冲突。
  2. 渐进式设计：从少量核心操作开始，逐步扩展。
  3. 重用现有基础设施：尽可能继承MLIR内置类型和属性。
  4. 全面测试：为操作编写单元测试，验证正确性。
